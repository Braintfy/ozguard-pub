# Глобальные правила ChatGPT/Codex (все проекты)

Этот файл — основная направляющая работы ChatGPT/Codex. Применять в любом проекте, если пользователь не дал более свежие или более конкретные инструкции.

## Коммуникация
- Лаконично, без воды, без "Отлично!", "Конечно!", "Вы правы!"
- Отвечать на языке пользователя; для этого пользователя по умолчанию — русский.
- При неясности, которая может привести к неправильному действию, уточнить ПЕРЕД выполнением.
- Анализировать запрос буквально, не додумывать скрытую задачу.
- Если есть несколько похожих элементов (кнопка, компонент, файл, ветка логики) — уточнить какой именно, если контекст не позволяет безопасно выбрать.
- Если работа уже понятна, действовать, а не останавливаться на плане.

## Жёсткие ограничения
- **Git**: НИКОГДА не делать `git add`, `git commit`, `git push`, `git reset`, `git checkout --` без прямой просьбы пользователя. По умолчанию только писать команды пользователю.
- **БД**: НИКОГДА не запускать миграции, push, seed, destructive SQL/DB-команды без прямой просьбы пользователя. По умолчанию только давать команды.
- **Файлы проекта**: читать перед изменением; править точечно; не трогать чужие/несвязанные изменения.
- **Скрипты для правок**: не использовать Python/bash/sed для ручной модификации файлов проекта, если можно применить `apply_patch`. Анализ через shell — можно.
- **Интерактивные команды**: если команда нужна и есть неинтерактивный режим, использовать `-y`, `--yes`, `--force` только когда это безопасно и не разрушительно.
- **Destructive-команды**: `rm`, `git reset --hard`, массовые удаления, force-перезаписи — только после явного разрешения.
- **Секреты**: не печатать ключи, токены, приватные значения в ответе; при деплое/публикации проверять, что секреты не попали в публичные артефакты.

## Процесс работы

### Начало задачи
1. Прочитать `CHATGPT.md` / `AGENTS.md`, если есть.
2. Поднять MemPalace-контекст, если задача существенная:
   ```bash
   mempalace wake-up --wing <project>
   ```
3. Если wing неизвестен — `mempalace status`, затем выбрать wing по имени проекта.
4. Прочитать `MEMORY.md`, `project.md`, `README.md`, `CHANGELOG.md`, если они есть и относятся к задаче.
5. Проанализировать запрос, определить затронутые файлы.
6. При сложной задаче — коротко озвучить план перед правками.

### Выполнение
1. Читать файлы ПЕРЕД изменением.
2. Искать через `rg` / `rg --files`.
3. Делать точечные правки, один логический участок за раз.
4. Использовать существующие паттерны проекта, а не придумывать новую архитектуру без необходимости.
5. После изменений запускать релевантную проверку: build, lint, test, `node --check`, typecheck или проектный аналог.
6. Если проверка невозможна, явно сказать почему.

### Завершение задачи
1. Обновить `CHANGELOG.md`, если изменение пользовательское/релизное или проект требует changelog.
2. Обновить `CHANGELOG-USER.md`, если проект ведёт пользовательский changelog.
3. Обновить `MEMORY.md`, `project.md` или MemPalace, если появилась важная долгосрочная информация.
4. Дать git-команды текстом, если пользователь ожидает commit/push:
   ```bash
   git add -A
   git commit -m "тип: описание"
   git push origin main
   ```
5. В финале кратко перечислить: что изменено, где, чем проверено, что осталось сделать.

## CHANGELOG формат

```markdown
## [X.Y.Z] - YYYY-MM-DD
### Added / Changed / Fixed / Removed
- **Компонент** — описание (1-2 строки, технические термины, пути файлов)
```

Версии: MAJOR.MINOR.PATCH. Если в проекте есть свой формат версий — соблюдать проектный формат.

## CHANGELOG-USER.md

Если проект ведёт отдельный пользовательский changelog, обновлять его вместе с техническим changelog.

Формат для OZGuard:
```markdown
## 🆕 Версия X.Y.Z — DD месяца YYYY

### Что стало лучше
- 🔧 **Короткий заголовок пункта.** Объяснение одним предложением простым языком.
- 🚀 Ещё один пункт...

### (опционально) Контекст — раньше/теперь
**Раньше:** описание проблемы.
**Теперь:** как работает сейчас.

*Опциональная курсивная заметка внизу релиза (косметика / требует внимания / BETA-предупреждение).*
```

Правила текста:
- Короткие предложения, активный залог.
- Без внутренних имён функций, storage-ключей и файлов, если changelog предназначен клиентам.
- Для важных фиксов — сценарий «Раньше / Теперь».
- Если фича BETA — явно сказать это и рекомендовать малый тест.

## MemPalace — долгосрочная память

**Установлен**: MemPalace v3.3.3, Python 3.11, ChromaDB локально.  
**Путь**: `/Users/firay/.mempalace`  
**Palace**: `/Users/firay/.mempalace/palace`  
**Приватный режим**: локальная база, без API-ключей.

### Архитектура

```text
Wings (проекты/люди) -> Rooms (темы) -> Closets (сводки/индекс) -> Drawers (оригиналы)
```

### Основные команды

```bash
mempalace status
mempalace wake-up --wing <project>
mempalace search "ключевой вопрос" --wing <project>
mempalace mine <project-dir> --wing <project>
mempalace mine <conversation-dir> --mode convos --wing <project>
mempalace mcp
```

### Как использовать в начале работы

1. Определить project wing:
   - обычно basename репозитория в нижнем регистре;
   - для OZGuard — `ozguard`.
2. Выполнить:
   ```bash
   mempalace wake-up --wing ozguard
   ```
3. Для конкретных рисков искать точечно:
   ```bash
   mempalace search "license activation formatter" --wing ozguard
   mempalace search "CHANGELOG-USER dual changelog" --wing ozguard
   ```
4. Если CLI в Codex sandbox получает `readonly database` или `No palace found`, повторить с elevated filesystem access: Chroma может открывать SQLite/HNSW с write-lock даже для чтения.

### Как сохранять новую память

Если появилась важная долгосрочная информация:
- Для локального проекта: обновить `MEMORY.md` / `project.md`, если такие файлы есть.
- Для MemPalace: использовать `mempalace mine` после обновления файлов памяти или добавить запись через MCP, если MemPalace MCP доступен.
- Не сохранять шум: одноразовые команды, временные ошибки, уже исправленные мелочи без будущей ценности.

### MCP интеграция

Если MemPalace MCP доступен, предпочитать MCP-инструменты:
- `mempalace_search`
- `mempalace_status`
- `mempalace_list_wings`
- `mempalace_add_drawer`
- `mempalace_diary_write`

Если MCP недоступен, использовать CLI fallback.

Состояние и логи:

## Агенты и параллельная работа

Codex sub-agents использовать только если пользователь явно попросил агентов, делегирование или параллельную агентную работу.

Если пользователь попросил агентов:
- Делить задачи на независимые, хорошо ограниченные части.
- Не отдавать агенту блокирующую задачу, если следующий локальный шаг зависит от её результата.
- Для code-change agents назначать явные зоны ответственности по файлам.
- Не запускать несколько агентов на один и тот же файл без необходимости.

Если пользователь не просил агентов:
- Работать самостоятельно.
- Использовать обычные shell/read/search операции.

## Frontend/UI работа

- Строить реальный рабочий экран, не landing page, если пользователь просит app/tool/game.
- Сохранять стиль существующего проекта.
- Проверять, что текст не налезает, элементы не перекрываются, мобильный вид не ломается.
- Использовать существующую icon library, если она есть.
- После существенных UI-изменений предлагать или запускать проверку в browser-use, если это уместно.
