## [5.8.1] - 2026-04-03
### Fixed (CRITICAL)
- **Жалобы: чат покупателя вместо поддержки** — `findSellerTab()` выбирал любую вкладку seller.ozon.ru, включая `group=customers_v2`. Теперь строгий приоритет: `group=support_v2` → messenger без group= → любая вкладка (с последующим redirect)
- **Жалобы: `ensureSellerChatPage` не перенаправлял** — URL с `/app/messenger?group=customers_v2` проходил проверку `includes('/app/messenger')`. Теперь проверяется именно `group=support_v2`
- **Жалобы: параллельная вкладка с покупателем** — добавлена проверка URL перед каждым шагом `supportProcessStep()`: если URL содержит `group=customers`, бот автоматически перенаправляет на `support_v2`
- **CPU freeze / зависание браузера** — рекурсивные вызовы `supportProcessStep()` (~30 мест) заменены на итеративный цикл `supportProcessLoop()` с `MAX_STEP_ITERATIONS=200`, `await delay(50)` между шагами для yield к event loop, аварийная остановка при превышении лимита
- **Зацикливание `in_progress`** — бот отправлял артикул в чат покупателя (не в поддержку), получал `in_progress` бесконечно. Исправлено валидацией URL + strict tab selection

## [5.6.0] - 2026-03-24
### Fixed (CRITICAL)
- **SKU пропуск / зацикливание на `ready_for_next`** — при перезапуске бота на существующем чате или после обработки предыдущего SKU, новый SKU видит `ready_for_next` и должен кликнуть «Пожаловаться на другой товар». Раньше бот либо помечал SKU как «обработанный» без действий, либо ждал смену фазы бесконечно. Fix: smart handler сразу кликает кнопку цикла (до 3 попыток), верифицирует смену фазы, и только при неудаче — ошибка. Стандартная защита от зацикливания пропускает этот случай (smart handler управляет сам)
- **`getMessageText()` стриппил текст бота** — `cursor:pointer` проверка удаляла текст из крупных контейнеров сообщений (вся строка бота вычиталась). Fix: стриппятся только мелкие элементы (height < 50px, width < 500px, без блочных дочерних, < 8 слов) — реальные quick-reply кнопки
- **`input_ready` fallback слишком агрессивный** — срабатывал через 2 повтора (6с), теперь через 3 повтора (9с) + расширены паттерны запроса артикула (`номер товара`, `укажите товар`, `sku`, `отправьте артикул`)

### Added
- **Smart Self-Correction System** — интеллектуальная система самокоррекции бота жалоб:
  - **Stale state detection** — распознаёт устаревшую фазу от предыдущего SKU, ждёт обновления страницы (до 5 попыток × 3с), затем пробует кликнуть кнопку цикла
  - **SKU verification** — перед пометкой «обработано» проверяет, что бот упомянул текущий SKU / содержит ключевые фразы результата
  - **Phase transition verification** — после клика «Пожаловаться на другой» проверяет, что фаза действительно сменилась с `ready_for_next`, добавляет доп. ожидание если нет
  - **Retry кнопки цикла** — при ошибке «Кнопка цикла не найдена» ждёт 5с и пробует повторно
  - **Подробный `[SMART]` лог** — все действия самокоррекции логируются с префиксом `[SMART]` для диагностики

### Fixed
- **API URL mismatch** — расширение вызывало `POST /api/license/activate` (action в URL path), а сервер ожидает `POST /api/license` с `{action, key, fingerprint}` в body. Все вызовы теперь идут на единый endpoint
- **VERIFY_INTERVAL** — исправлен интервал фоновой проверки: 24ч (было 12ч) для соответствия серверному API

### Changed
- **`getMessageText()`** — полностью переписана логика фильтрации: вместо удаления ВСЕХ cursor:pointer элементов, фильтрует по размеру, структуре и длине текста
- **`input_ready` → 3 повтора** — увеличен порог fallback с 2 до 3, добавлен вывод текста бота в лог для диагностики
- **Комментарии** — убраны упоминания чувствительной информации из кода и CHANGELOG

## [5.2.0] - 2026-03-20
### Changed (CRITICAL)
- **`findSendButton()` — по SVG path** — кнопка отправки и кнопка прикрепления имеют одинаковый класс `om_17_a1`. Теперь различаются по SVG path: стрелка отправки = `M5.086`, скрепка = `M11.055`. Fallback: последняя SVG-кнопка в контейнере textarea
- **`detectPhase()` — по последнему сообщению бота** — полностью переписан. Использует атрибут `[is-mine]` для чтения истории чата (is-mine="" = бот, is-mine="mine" = пользователь). Фаза определяется по ПОСЛЕДНЕМУ сообщению бота + наличию ответа пользователя. Это позволяет боту понимать, на каком шаге остановился чат
- **`in_progress` с detail** — если последнее сообщение от пользователя (бот ещё не ответил), возвращается `in_progress` с detail: `article_sent_waiting` или `file_sent_waiting`. Service-worker корректно обновляет `item.step` из detail

### Added
- **`readChatHistory()`** — новая функция, читает все сообщения чата по `[is-mine]` атрибуту с определением отправителя и типа (текст/файл)
- **`getLastMessages()`** — возвращает последнее сообщение бота, пользователя и общее для точного определения фазы
- **Авто-продолжение с середины чата** — бот корректно определяет фазу даже если чат уже был начат ранее: если бот попросил артикул и пользователь уже отправил → `in_progress`, если бот попросил файл → `waiting_attachment`

### Fixed
- **Кнопка отправки путалась с кнопкой прикрепления** — обе имели одинаковый класс и обе содержали SVG. Бот кликал по скрепке вместо стрелки → файл не отправлялся
- **Phase detection на старых сообщениях** — ранее `readLastMessages` собирал ВСЕ видимые текстовые блоки, включая старые сообщения о «скрыли товар» от предыдущих артикулов. Теперь используется `[is-mine]` для точного чтения последнего сообщения бота
